## Findings

### 1. CLI `core status` disagrees with API/systemd

`sudo shilka core status` returned:

```json
{
  "Running": false,
  "PID": 0,
  "Version": "sing-box 1.13.12",
  "Uptime": 0
}
```

At the same time:

- `systemctl status shilka.service` showed a running managed `sing-box` child;
- `GET /api/core/status` returned `running: true` with the correct PID.

This looks like a CLI process-manager detection issue, not an actual core
startup failure.

### 2. Node self-probe cannot validate self-signed panel TLS

The temporary self-node was created successfully with `allow-private`, but
`sudo shilka node probe` marked it `offline` with:

```text
x509: certificate signed by unknown authority
```

`sudo shilka node sync` failed with the same TLS verification error.

The node client currently has no CLI or model option for insecure/self-signed
TLS verification. For self-node smoke with the current release, either use a
trusted certificate/reverse proxy path or add explicit controlled support for
self-signed verification in node client configuration.

### 3. Release `1.7.0` links response uses `link`, not local `shareLink`

`GET /api/clients/{id}/links` returned a payload with:

```json
{
  "link": "...",
  "subscription": "..."
}
```

The current local frontend DTO names the share link field `shareLink`. Align the
handler and DTO before the next release, or keep backward-compatible decoding in
the frontend.

### 4. Release metadata is not semantic in two places

Observed release responses:

- `GET /api` returned version `0.0.0`.
- `GET /api/node/v1/status` returned `panelVersion: "shilka"`.

This does not block installation, but it weakens release smoke evidence and
node inventory quality.

## Follow-up Plan

1. Fix CLI `core status` so it reports the same running core that API/systemd
   observe when the panel owns the managed sing-box subprocess.
2. Decide how node sync should work with self-signed panel TLS:
   - preferred production path: trusted TLS through reverse proxy or ACME;
   - controlled smoke/dev path: explicit per-node `skip_tls_verify` or trusted
     CA configuration, never silent global insecure behavior.
3. Align client links response naming:
   - either change backend response to `shareLink`;
   - or update frontend DTO/runtime handling to accept both `link` and
     `shareLink`.
4. Fix release version metadata so `/api` and `/api/node/v1/status` expose the
   real panel version.
5. For the next production-grade smoke, publish a new release from the merged
   local changes and rerun the installer from the canonical raw GitHub URL
   instead of copying the local script manually.
6. If the server must keep Shilka long-term, replace self-signed public access
   with a reverse proxy or trusted certificate path and record the chosen
   operational URL outside the repository.

Shilka remains installed and running on the VPS in no-downtime direct high-port
